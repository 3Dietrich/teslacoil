## A. Bugfixes (klein, hoher Wert – zuerst)

### A1 · Filter HP/BP reparieren + Cutoff-Env bipolar
**Befund @dpa:** HP klingt nicht wie HP, BP klingt wie LP. **Ursache:** In der
letzten Runde wurde die Cutoff-Env bei HP/BP *abwärts* gelenkt (Ziel 20 Hz) →
beim Trigger fällt der Cutoff Richtung 20 Hz → HP@20 Hz ≈ alles durch (klingt
offen), BP@20 Hz ≈ tiefe durch (klingt wie LP). Das erklärt BEIDE Befunde.

**Umsetzung in [js/engine/TeslaEngine.js](../js/engine/TeslaEngine.js) `_onTrigger`:**
1. Die typabhängige Abwärts-Umkehr WIEDER ENTFERNEN. Es gibt nur EINE Env-Richtung
   für alle Typen: die Env moduliert den Cutoff, Vorzeichen/Betrag kommen aus dem
   Regler (s.u.). Kein `target = (HP||BP) ? 20 : maxF` mehr.
2. **`lpEnv` wird bipolar −1..+1** (State-Default bleibt 0.5). Semantik:
   `peak = clamp(effCutoff · (maxF/effCutoff)^(lpEnv·depth), 20, maxF)`
   – bei `lpEnv>0` steigt der Cutoff (heller/öffnet bei LP), bei `lpEnv<0` sinkt er
   (Richtung 20 Hz), bei `0` keine Env. Das gilt für **alle** Typen identisch; der
   User wählt das „Vorzeichen" selbst. Formel so schreiben, dass negatives Exponent-
   Vorzeichen den Cutoff korrekt nach unten zieht (Exponent = `lpEnv·depth`, kann
   negativ sein → `(maxF/effCutoff)^negativ` < 1 → Cutoff sinkt). Ergebnis auf
   `[20, maxF]` klemmen.
3. Die Bedingung `if (env > 0)` in `_onTrigger` wird zu `if (lpEnv !== 0)`
   (Betrag zählt, nicht nur positiv). `_applyFilter` setzt den statischen Cutoff
   weiterhin nur, wenn `lpEnv === 0 && lpKeyTrack === 0`.

**[js/app.js](../js/app.js) KNOBS:** `lpEnv` → `{ min: -1, max: 1, curve: 'linear', decimals: 2 }`.
**GOTCHA (@dpa ausdrücklich):** Die feste Range −1..+1 muss in KNOBS stehen und
darf NICHT durch einen gespeicherten Snapshot-/Meta-Wert beschnitten werden – ein
Snapshot mit lpEnv=−1 darf den Regelbereich nicht auf −1 „einfrieren". (Range kommt
aus KNOBS/`_defaultMeta`, der Wert aus dem State – nie den Wert zur neuen max/min machen.)

**Verifikation:** `dsp_filter_check.mjs`-Muster (110 Hz + 6000 Hz Mischsignal durch
`LadderCore`): HP muss 110 Hz stark dämpfen, 6000 Hz durchlassen; BP muss beide
Flanken dämpfen (Peak bei Cutoff). Cutoff-Default für den Test 4000 Hz. Dann @dpa
live bypass-vergleichen lassen.

### A2 · KeyTrack auch bei Env ≠ 0
**Befund @dpa:** Bei `lpEnv=0` bekommt JEDER Step die gespielte Freq auf den Cutoff
(via `setCutoffAt`) – „genau so gut". Er will das auch mit Env ≠ 0.
**Umsetzung `_onTrigger`:** Den `setCutoffAt(effCutoff)`-Pfad (KeyTrack-Basiswert
pro Note) auch dann laufen lassen, wenn `lpEnv ≠ 0`, ABER nur für Steps OHNE
Env-Trigger (z. B. Filter-Seq-Steps = 0). Logik:
```
if (lpEnv !== 0 && doTrig) {
    triggerEnv(time, effCutoff, peak, ...);   // Env von der KeyTrack-Basis aus
} else if (kt > 0 && noteOn) {
    setCutoffAt(time, effCutoff);             // KeyTrack-Basis (auch bei Env≠0)
}
```
So ist `effCutoff` (= `lpCutoff + kt·_lastFreq`) IMMER die Basis pro gespielter
Note; die Env sitzt bei getriggerten Steps oben drauf. Kollision zweier
Automationsquellen vermeiden (nie im selben Step beide auf `cutoff` schreiben – die
if/else-if-Struktur garantiert das).
**Verifikation:** logic-Test o. Node-Harness: bei zwei verschiedenen Freqs unter-
schiedliche effCutoff-Basis; @dpa live.

### A3 · Amp-Env „Dyn" bipolar −1..+1
**Ist:** `ampSeqDyn` 0..1 (linear→v²). **Soll @dpa:** −1 [alle Steps 75 %] · 0
[lineare Ausgabe] · +1 [100 % „Kontrast", v²].
**Umsetzung `_onTrigger`** (Amp-Seq-Velocity-Kurve):
```
const dyn = s.get('ampSeqDyn');            // −1..+1
if (dyn >= 0) ampGate = g*(1-dyn) + g*g*dyn;          // linear → v²
else          ampGate = g*(1+dyn) + 0.75*(-dyn);       // linear → konstant 0.75
```
(`g` = roher Step-Wert 0..1.) **KNOBS `ampSeqDyn`** → `{ min:-1, max:1, decimals:2 }`,
Default 0. Gleiche Range-GOTCHA wie A1 (Snapshot darf Range nicht klemmen).

## B. Umbauten

### B4 · „Voices" → Poly-Limit (Missverständnis korrigieren)
**@dpa meinte NICHT Unison**, sondern die überlappenden, nacheinander getriggerten
Oszillatoren (bei langsamer Amp-Env/langem Attack laufen aufeinanderfolgende Trigger-
Voices gleichzeitig = natürliche Polyphonie).
**Umsetzung:**
1. `oscVoices`/`oscSpread` + zugehörige Unison-Logik in
   [js/audio/SquareOsc.js](../js/audio/SquareOsc.js) `trigger()`/`_spawnVoice` **entfernen**
   (zurück auf eine Voice pro Trigger), KNOBS/GROUPS/State-Defaults dazu entfernen.
2. Stattdessen **`polyMax`** (State-Default z. B. 8, Range 1..8) einführen: die
   maximale Anzahl gleichzeitig KLINGENDER Trigger-Voices. In `SquareOsc`: `_active`
   ist bereits ein Set laufender Voices → beim `trigger()` wenn `_active.size >= polyMax`,
   die ÄLTESTE Voice sanft stehlen (schnelles Release ~5 ms, dann stop/disconnect).
   Voice-Stealing FIFO. So kann @dpa mit langen Env/Len bewusst Cluster bauen ODER
   die Polyphonie hart begrenzen.
3. UI: Knob `polyMax` in Gruppe „Audio-Osz" (oder „Envelope"). Sound-Param → im Snapshot.
**Erklärung an @dpa (in Antwort, nicht Code):** die „2 nacheinander triggernden
OSC" bei langer Amp-Env sind kein Bug – es sind überlappende Voices, weil die Env
länger dauert als das Trigger-Intervall. polyMax macht das jetzt steuerbar.

### B5 · Metronom-Filter: Cutoff-„Quant" an BaseFrq
**Soll @dpa:** Toggle `metroCutoffQuant`. Aus = Cutoff-Knob wie bisher. An =
Cutoff-Knob wird ersetzt durch einen Oktaver-Knob `metroCutoffOct` (Range −2..+6,
step 1) mit **`cutoff = BaseFrq · 2^oct`**.
**Umsetzung:**
- State: `metroCutoffQuant: false`, `metroCutoffOct: 0`.
- [js/engine/TeslaEngine.js](../js/engine/TeslaEngine.js) `_rebuildMetro`: wenn Quant an,
  `cutoff = clamp(this.baseFreq · 2^metroCutoffOct, 20, 18000)` statt `metroCutoff`.
  **WICHTIG:** Bei Quant an muss der Metro-Klick bei BaseFrq-Änderung neu gerendert
  werden → `_rebuildMetro` auch bei den BaseFrq-Keys (`baseSrc/baseHz/baseNote/baseOct/
  tempoOct/bpm`) aufrufen, wenn `metroCutoffQuant` an ist (analog Test-Ton-Nachführung).
- UI: Toggle + Oktaver-Knob in Gruppe Metronom; `metroCutoff`-Knob nur sichtbar wenn
  Quant AUS, `metroCutoffOct` nur wenn Quant AN (Sichtbarkeitslogik wie updateMetroVisibility).

### B6 · Takt: BaseFrq-Oktaver (= gleicher Mechanismus wie B5, @dpa bestätigt)
**@dpa:** „Base Freq fernsteuerung mit 'oct'" = *dasselbe wie der Metronom-Cutoff-
Oktaver (B5)*: ein Oktaver-Knob **`baseOctave` Range −2..+6, step 1**, der die
effektive BaseFrq mit **`· 2^baseOctave`** transponiert – quellenübergreifend (gilt
für ALLE `baseSrc`, nicht nur Ton-Modus).
**Umsetzung:**
- State: `baseOctave: 0`.
- [js/engine/TeslaEngine.js](../js/engine/TeslaEngine.js) `get baseFreq`: das Ergebnis
  aller Quellen zusätzlich `· Math.pow(2, baseOctave)` (ganz am Ende, nach dem switch).
- UI: Knob `baseOctave` in Gruppe „Takt" oder „Base-Frq".
- **Fernsteuerung:** die bestehende Pfeiltasten-Logik (aktuell `ArrowUp/Down` →
  `baseOct` nur im Ton-Modus) so erweitern, dass ↑/↓ in JEDEM Modus `baseOctave`
  (−2..+6) steuern. (Ton-Modus-spezifisches `baseOct`/Links-Rechts bleibt erhalten.)
- Sound-Param → im Snapshot. `baseFreq`-Nachführungen (Test-Ton A? , Metro-Quant B5)
  greifen automatisch, da sie den `baseFreq`-Getter nutzen.
