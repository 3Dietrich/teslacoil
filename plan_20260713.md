## Vorab: die zu behebende Isolation-Kante

In [freezeGroup](js/app.js#L964) (Z. 967) werden beim Einfrieren `display:none`-Controls
übersprungen (`if (u.offsetParent === null) return;`) → sie bekommen keine `ctrlPos` und springen
beim Wiedereinblenden an den Anfang / überlagern sich. **Block A löst das an der Wurzel:** im
e-mode immer alles zeigen → nichts ist `display:none` → jedes Control bekommt eine Position.

## Block A — e-mode Sichtbarkeit + Isolation-Fix 🟢

- Im e-mode (`arranging === true`) die on/off-Stellung **optisch ignorieren**: alles anzeigen,
  was anzeigbar ist. Verlassen des e-mode → normale Sichtbarkeit wieder herstellen.
- Dadurch entfällt die Freeze-Kante (siehe oben) automatisch.
- **click-Springen unterbinden:** manche Elemente „springen nach unten", wenn man sie anklickt.
  Beim Selektieren im e-mode kein Layout-Sprung / kein `scrollIntoView`.

**Anker:** `setArranging` [Z. 1050](js/app.js#L1050), `setSelected` [Z. 672](js/app.js#L672),
Sichtbarkeits-Helfer `setVis` [Z. 1212](js/app.js#L1212), `VIS_TOGGLE_KEYS` [Z. 355](js/app.js#L355),
Filter-Visibility-Refresh ~[Z. 1225](js/app.js#L1225), `freezeGroup` [Z. 964](js/app.js#L964).

## Block B — Filter-Semantik 🟡

- **„aktiv" aus** (`filterEnabled=false`): bis auf **Type** alles ausblenden. Aktuell bleiben
  Glide und Sequenzer sichtbar → auch die verstecken.
- **„Env-Trig"** (ehemals „Seq", Key `filterSeqEnabled` [Z. 105](js/app.js#L105)) **dreistufig**
  statt Toggle:
  - `off`: Env aus → Sequenzer, Env, Att, Dec ausgrauen/ausblenden
  - `each`: Env bei jedem Trigger volle „Env"-Amount → Sequenzer ausblenden
  - `Seq`: wie bisher (Sequenzer steuert Env-Trigger/-Depth pro Step)
  - Neuer State-Key (z.B. `filterEnvTrig: 'seq'`), **Migration** von altem `filterSeqEnabled`
    (true→'seq', false→'off'). Engine-Trigger-Verzweigung in
    [TeslaEngine.js](js/engine/TeslaEngine.js) anpassen; Logik-Test ergänzen.

**Anker:** `TOGGLES`/`SELECTS`-Defs [Z. 104–106](js/app.js#L104), Filter-Gruppe
[Z. 135](js/app.js#L135), Filter-Visibility ~[Z. 1225](js/app.js#L1225).

## Block G — Standalone: Header/Takt/Metronom 🔴/🟡 (eigenes Artefakt)

Erst als **extra HTML** (eigenständig, kein Risiko für die Haupt-App). Mehrere Commits:

- **Start/Stop-Settings:** Button-Größe; [MIDI: „learn"-Button mit Klassen-Filter
  (Note/Control/Ch 0-all/1-16 …), reset, gelernte Anzeige] 🔴 = eigenes Subsystem (Web-MIDI).
- **Tab-Tempo (in „Takt"):** „Tab [Metronom-Icon]"-Tempo, „!go"-Button, „wait"-Control
  (ab wann Pause). 🟡 = anspruchsvollster Algorithmus, eigener Durchgang + Unit-Tests:
  - **Mode 1** (gleichbleibendes Tempo): erste 2 Klicks messen das Timing; bei weiteren Klicks
    im ±15 %-Fenster intelligent das Timing über **alle** Klicks mitteln. Bei 190–210 %
    jeder 2. (≈300 % jeder 3., ≈400 % jeder 4. …) berechnen. Annahme: man klickt möglichst
    synchron auf etwas Gehörtes → **Gesamtlänge aller Klicks** als Teil des neuen Tempos einbeziehen.
  - **Mode 2** (sich ständig änderndes Tempo): wie Mode 1, nur wird ab dem 3. Klick „!go"
    mitgetriggert.
